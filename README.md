## 2. Pin Assignment

| Pin | Signal | Direction | Description |
|-----|--------|-----------|-------------|
| PA0 | STEP_X | Output | X-axis step pulse — toggled in TIM2 ISR |
| PA1 | STEP_Y | Output | Y-axis step pulse — toggled in TIM2 ISR |
| PA4 | DIR_X  | Output | X-axis direction — HIGH = positive |
| PA5 | DIR_Y  | Output | Y-axis direction — HIGH = positive |
| PA11 | USB D− | USB FS | Hardware USB |
| PA12 | USB D+ | USB FS | Hardware USB |

> **Important:** PA0 and PA1 are plain GPIO outputs, **not** TIM2 alternate-function
> pins. Step pulses are generated by direct `BSRR`/`BRR` register writes inside
> the ISR. Do not configure these pins as TIM2 AF in CubeMX.

## 3. Driver Wiring

### Per-Axis Connections (repeat for X and Y)

```
STM32          Driver (A4988 / DRV8825)      Motor
------         -------------------------      -----
PA0 ─────────► STEP                          
PA4 ─────────► DIR                           
GND ──────────  GND                          
3.3V ─────────  VDD (logic)                  
               VMOT ◄──── 12–36 V PSU        
               GND  ◄──── PSU GND            
               1A ────────────────────────► Coil A+
               1B ────────────────────────► Coil A−
               2A ────────────────────────► Coil B+
               2B ────────────────────────► Coil B−
```

### Critical Wiring Rules

1. **EN pin** — tie to GND permanently. If left floating the driver is
   disabled and the motor will not move.

2. **100 µF decoupling cap** — place across VMOT and GND on each driver,
   as close to the board as possible. Without it, motor back-EMF spikes
   can destroy the driver IC.

3. **Common GND** — STM32 GND, driver logic GND, and motor PSU GND must
   all share the same reference point.

4. **Current limit** — set the potentiometer on the driver to match your
   motor's rated current before running. Too high = overheating;
   too low = stall.

5. **DIR setup time** — A4988 requires 200 ns, DRV8825 requires 650 ns.
   The firmware inserts a 400 µs hold (4 ISR ticks) after every direction
   change before the first step pulse, covering both drivers with margin.

## 4. System Architecture

```
USB Host (PC Terminal)
        │
        │ USB Full-Speed (12 Mbps)
        ▼
[USB ISR — CDC_Receive_FS]
        │  pushes bytes into RX ring buffer (256 bytes)
        │  NO parsing here — ISR context only
        ▼
[Main Loop — GCode_Poll()]
        │  USB_CDC_DrainRX() → GCode_PutChar() → line buffer
        │  on '\n': execute_line() → do_move() → Stepper_MoveTo()
        │  USB_CDC_TxPoll() → drains TX ring → CDC_Transmit_FS()
        ▼
[TIM2 Update ISR — every 100 µs]
        │  for each axis: if now >= s_next_step[axis] → fire step
        │  STEP pin HIGH → count step → update ramp → schedule next
        │  STEP pin LOW after 10 µs pulse width
        ▼
[Stepper Driver — A4988/DRV8825]
        │  converts STEP/DIR signals to motor coil currents
        ▼
[Stepper Motor]
```

### Ring Buffer Architecture

Two ring buffers decouple the USB ISR from main-loop processing:

| Buffer | Size | Written by | Read by |
|--------|------|-----------|---------|
| RX ring | 256 bytes | USB ISR | Main loop (GCode_Poll) |
| TX ring | 512 bytes | GCode_Send | USB TxPoll / DataIn callback |

This prevents:
- Parsing or blocking inside the USB interrupt (would deadlock with TIM2)
- Passing stack-local buffers to USB DMA (causes garbled output)
- `HAL_Delay` spinning inside TX retry loops

## 5. Timer & Step Generation Theory

### Why plain GPIO instead of PWM?

TIM2 PWM mode generates pulses via hardware compare match but requires
the Update IRQ to count steps — these happen at different times causing
miscounts. Calling `HAL_TIM_Base_Start_IT` after `HAL_TIM_PWM_Start`
also resets HAL internal state causing unpredictable IRQ behaviour.

The solution: TIM2 runs as a plain up-counter. Step pulses are generated
by direct `BSRR`/`BRR` writes inside the ISR.

### Timer Configuration

```
Input clock:  48 MHz (PCLK1 × 2 on APB1)
Prescaler:    PSC = 47  →  48 MHz / 48 = 1 MHz (1 µs per tick)
Period:       ARR = 99  →  IRQ every 100 µs (10 kHz poll rate)
```

The 16-bit TIM2 counter (max 65535) is extended to 32 bits by a
software `s_tick_base` variable that accumulates `+100` on every
IRQ overflow.

### Why PSC=47 is critical

Without a prescaler:
```
ARR = 48,000,000 / sps − 1
At 50 sps: ARR = 959,999  → OVERFLOWS 16-bit register → wrong speed
At 100 sps: ARR = 479,999 → OVERFLOWS                  → wrong speed
At 732 sps: ARR = 65,535  → fits (barely)
```

With PSC=47 (1 MHz tick):
```
ARR = 1,000,000 / sps − 1
At 50 sps:    ARR = 19,999  ✓
At 500 sps:   ARR = 1,999   ✓
At 2000 sps:  ARR = 499     ✓
At 20000 sps: ARR = 49      ✓
```

### Absolute Tick Scheduler

Each axis independently tracks `s_next_step[axis]` — the 32-bit
microsecond timestamp of its next step:

```c
// After each step, schedule the next:
s_next_step[axis] = now_ticks() + (1,000,000 / cur_speed_sps);

// ISR check (every 100 µs):
if (now_ticks() >= s_next_step[axis])
    axis_do_step(axis);
```

This gives truly independent step rates on one timer with no
approximation errors. Unlike a tick-divider approach, the schedule
automatically adapts every step as `cur_speed_sps` changes during ramping.

### Step Pulse Timing

```
STEP pin:  ___/‾‾‾‾‾‾‾‾‾‾\___________
                ← 10 µs →
           ^                ^
           Rising edge       Falling edge
           count step        (s_pulse_end check)
```

Pulse width = 10 µs (PULSE_TICKS = 10 at 1 µs/tick).
A4988 minimum pulse width = 1 µs. DRV8825 = 1.9 µs. Both covered.

## 6. Trapezoidal Ramp Theory

### Why smooth ramping matters

Stepper motors are synchronous — if the step rate exceeds what the
rotor can physically follow, it loses synchronisation and stalls.
Smooth acceleration lets the rotor keep up. Abrupt speed changes
cause resonance (noise) at certain frequencies.

### Bresenham Kinematic Formula

The ramp uses the kinematic equation `v² = v₀² + 2·a·Δs` where
each step is Δs = 1:

```
Accel:  v_new = sqrt(v_old² + 2·a)
Decel:  v_new = sqrt(v_old² − 2·a)
```

This gives **constant acceleration in physical time** — equal change
in speed per unit distance — which is exactly what the motor expects.

The naive formula `v_new = v_old + a/v_old` (linear per-step addition)
gives large speed jumps at low speeds and tiny increments at high
speeds, causing the motor to lurch and resonate.

### Ramp Phases

```
Speed
  ^
  |          _______________  ← cruise_speed_sps
  |         /               \
  |        / ACCEL    DECEL  \
  |       /                   \
  |______/                     \______
  └─────────────────────────────────► Steps
  start                             end
  MOTOR_ACCEL  MOTOR_CRUISE  MOTOR_DECEL
```

| Phase | Condition to enter | Speed update |
|-------|--------------------|-------------|
| ACCEL | Always at start | `v = sqrt(v² + 2·a)` each step |
| CRUISE | `v >= cruise_speed` | constant |
| DECEL | `remaining_steps <= decel_distance` | `v = sqrt(v² − 2·a)` each step |

### Deceleration Distance

The firmware calculates how many steps are needed to decelerate from
cruise speed down to minimum speed:

```
decel_steps = (v_cruise² − v_min²) / (2 × accel)
```

When `remaining_steps` equals this value, deceleration begins.
This ensures the motor always reaches exactly the target position
at minimum speed.

### Triangle Profile (Short Moves)

For moves where the total steps are less than `2 × decel_steps`,
the motor never reaches cruise speed. It accelerates to a peak
then immediately decelerates:

```
peak_speed = sqrt(accel × total_steps)
```

This prevents over-speed on short moves.

### Default Ramp Values

| Parameter | Default | G-Code setting |
|-----------|---------|----------------|
| Max speed | 1000 sps | `$0=` (X), `$1=` (Y) |
| Acceleration | 5000 sps² | `$2=` (X), `$3=` (Y) |
| Min speed | 50 sps | fixed in firmware |
| Max allowed | 20000 sps | hardware limit |

With accel = 5000 sps² and cruise = 1000 sps:
- Accel distance = (1000² − 50²) / (2 × 5000) = **~100 steps**
- Decel distance = same = **~100 steps**
- A 200-step move uses the full trapezoid

## 7. Dual-Axis Simultaneous Motion

### How simultaneous motion works

`do_move()` in `gcode.c` starts both axes in the same main-loop
cycle with no wait between them:

```c
Stepper_MoveTo(AXIS_X, mx);   // arm X
Stepper_MoveTo(AXIS_Y, my);   // arm Y immediately after
wait_all();                   // wait for BOTH to finish
```

Both axes are armed within a few hundred nanoseconds of each other.
The TIM2 ISR fires every 100 µs and independently checks each axis's
`s_next_step` timestamp — both axes can fire steps in the same ISR
call if their schedules coincide.

### Linear Interpolation (G1)

For coordinated linear motion, the per-axis speeds are scaled so
both axes finish at the same time:

```
Dominant axis (more steps):  speed = F (commanded feed rate)
Subordinate axis (fewer steps):  speed = F × (steps_minor / steps_major)
```

**Example:** `G1 X1000 Y600 F800`
- X has more steps → X runs at 800 sps
- Y speed = 800 × 600/1000 = **480 sps**
- Both finish in 1000/800 = **1.25 seconds**
- Path is a straight line from start to (1000, 600)

### G0 Rapid Move

`G0` moves both axes at their configured max speed independently.
The shorter axis finishes first. This is correct rapid behaviour —
the path is not a straight line but speed is maximised.

```
G0 X1000 Y1000   → both axes at max speed simultaneously
G0 X1000 Y200    → X runs to 1000, Y runs to 200; Y finishes first
```

## 8. USB CDC Communication

### Connection

Connect the Blue Pill USB to your PC. It enumerates as a USB CDC
Virtual COM Port. Use any terminal:

| Terminal | Settings |
|----------|----------|
| PuTTY | 115200 baud (baud rate is ignored — USB CDC), 8N1 |
| Arduino IDE Serial Monitor | Any baud rate |
| CoolTerm | Any baud rate |
| Python `pyserial` | Any baud rate |

### Line Ending

Send commands terminated with `\n` (LF). Most terminals do this
with Enter. The firmware accepts `\n` only; `\r` is silently ignored.

### Real-Time Commands

These single characters are processed immediately — **no newline needed**:

| Char | Action |
|------|--------|
| `?`  | Status report |
| `!`  | Feed hold (immediate stop) |
| `~`  | Cycle start / resume |
| Ctrl-X (0x18) | Soft reset |

### Responses

| Response | Meaning |
|----------|---------|
| `ok` | Command accepted and completed |
| `error:msg` | Command rejected |
| `<state\|MPos:x,y\|WPos:x,y\|F:n>` | Status report |
| `HOLD` | Feed hold acknowledged |
| `RESUME` | Cycle start acknowledged |
| `ALARM` | Emergency stop (M112) |
| `[MSG:text]` | Informational message |

## 9. G-Code Command Reference

### Motion Commands

#### G0 — Rapid Move
```
G0 X<pos> Y<pos>
```
Moves both axes simultaneously at maximum speed. Path is not
guaranteed to be a straight line. Used for positioning moves
where speed matters more than path accuracy.

```
G0 X500 Y500      → move both axes to position 500
G0 X0             → move X to 0 only (Y stays)
G0 Y1000          → move Y to 1000 only (X stays)
```

#### G1 — Linear Feed Move
```
G1 X<pos> Y<pos> F<speed>
```
Coordinated linear move. Both axes start and finish together.
F sets the feed rate in steps/s. If F is omitted, the last
commanded feed rate is used.

```
G1 X1000 Y1000 F500   → diagonal move at 500 sps resultant
G1 X500 F300          → X-only move at 300 sps
G1 Y200 F800          → Y-only move at 800 sps
```

#### G4 — Dwell
```
G4 P<milliseconds>
```
Pause motion for P milliseconds. Motion before G4 completes
fully before the dwell begins.

```
G4 P500    → pause 500 ms
G4 P1000   → pause 1 second
```

#### G28 — Go Home
```
G28
```
Moves both axes to the work position origin (0, 0) at rapid speed,
then zeroes the machine position.

#### G90 — Absolute Mode (default)
```
G90
```
All subsequent X/Y values are absolute positions from the work origin.

```
G90
G1 X100    → move TO position 100
G1 X200    → move TO position 200 (travels 100 steps)
```

#### G91 — Relative Mode
```
G91
```
All subsequent X/Y values are relative offsets from current position.

```
G91
G1 X100    → move 100 steps from current position
G1 X100    → move another 100 steps (now at +200 total)
G91
G1 X-50    → move back 50 steps
```

#### G92 — Set Work Position
```
G92 X<pos> Y<pos>
```
Defines the current machine position as the specified work coordinate.
Does not move the motor. Used to set a reference without homing.

```
G92 X0 Y0     → define current position as work origin
G92 X100      → define current position as X=100 in work coords
```

## 11. Startup Auto-Move

On power-up, before accepting USB commands, the firmware automatically
runs a startup sequence on the X axis to verify motion is working:

```c
Stepper_SetSpeed(AXIS_X, 500);   // 500 steps/s
Stepper_SetAccel(AXIS_X, 800);   // 800 steps/s²
Stepper_MoveTo(AXIS_X, 200);     // move forward 200 steps
// wait...
Stepper_MoveTo(AXIS_X, 0);       // return home
// wait...
GCode_Send("ok X homing done\r\n");
```

After this completes, the system enters normal USB command mode.

To modify the startup move, edit the values in `main.c` after
`GCode_Init()`. To disable it, remove those lines entirely.

## 12. IRQ Priority Map

On Cortex-M3, **lower number = higher priority**.

| Interrupt | Priority | Reason |
|-----------|----------|--------|
| TIM2 Update | 0 (highest) | Step generation must never be delayed |
| USB (USB_LP_CAN1_RX0) | 1 | Below TIM2 so steps fire even during USB traffic |
| SysTick | 15 (lowest) | HAL tick — not time-critical |

**Why TIM2 must be higher than USB:**

The main loop spins `while(Stepper_IsAxisBusy())` waiting for the
TIM2 ISR to complete each move. If USB (priority 1) is servicing a
packet and TIM2 (priority 0) fires, TIM2 preempts USB and runs the
step. If the priorities were reversed, USB would block TIM2 and the
wait loop would spin forever → system hangs.

## 13. Tuning Guide

### Steps/mm Conversion

The firmware works in steps, not millimetres. To command a distance
in mm, multiply by your steps/mm:

```
steps/mm = (motor_steps_per_rev × microstepping) / lead_screw_pitch_mm

Example:
  Motor: 200 steps/rev
  Driver: 1/16 microstepping → 200 × 16 = 3200 steps/rev
  Lead screw: 8 mm/rev
  steps/mm = 3200 / 8 = 400 steps/mm

  To move 10 mm: G1 X4000 F2000
```

### Choosing Acceleration

Too low → slow ramp, wasted time.
Too high → motor stalls during acceleration.

Start with `$2=2000` and increase until the motor stalls, then
back off 20–30%.

With the Bresenham formula, ramp distance in steps:
```
accel_steps = (v_cruise² − v_min²) / (2 × accel)

Example: cruise=1000, min=50, accel=5000
  = (1000000 − 2500) / 10000 = ~100 steps
```

### Choosing Max Speed

Maximum reliable speed depends on:
- Motor rated current (higher = more torque at speed)
- Motor inductance (lower = faster response)
- Supply voltage (higher = better high-speed torque)
- Load / inertia

Start at `$0=500` and increase in steps of 200 until missed steps
appear, then back off 25%.

### Microstepping

Higher microstepping gives smoother motion and lower noise but
reduces torque at the same current. For a humanoid base:
- 1/8 or 1/16 microstepping is a good starting point
- Set on the driver via MS1/MS2/MS3 pins (see driver datasheet)

## 14. Troubleshooting

### Motor not moving

| Check | Fix |
|-------|-----|
| EN pin | Tie to GND on driver |
| Common GND | Connect PSU GND to STM32 GND |
| VMOT voltage | Must be within driver's rated range (8–35V for A4988) |
| Current limit | Adjust potentiometer on driver |
| DIR setup time | Already handled by firmware (400 µs hold) |
| TIM2 NVIC | Verify `HAL_NVIC_EnableIRQ(TIM2_IRQn)` is called in `HAL_TIM_Base_MspInit` |

### Motor moves only one direction

- DIR pin is not connected or is floating
- Check PA4 (DIR_X) and PA5 (DIR_Y) are physically wired to driver DIR input
- Verify common GND between STM32 and driver

### Motor noisy / vibrating

- Increase acceleration gradually (`$2=`, `$3=`)
- Reduce max speed (`$0=`, `$1=`)
- Increase microstepping on driver
- Add 100 µF capacitor across VMOT/GND if not already present
- Check motor current limit — too low causes resonance

### USB not responding after a few commands

- Firmware uses TX ring buffer (512 bytes) + USB TX poll
- If TX ring fills (too many rapid commands), bytes are dropped
- Send commands at human typing speed or with 50 ms delay between lines
- Check that `CDC_Transmit_FS` is not being called from ISR context

### System hangs after G0/G1 command

- Indicates TIM2 ISR is not firing
- Verify TIM2 NVIC priority = 0 and USB priority = 1 in source files
- Check `TIM2_IRQHandler` calls `Stepper_TIM_IRQHandler()` in `stm32f1xx_it.c`
- Rebuild clean (Project → Clean in CubeIDE)

### `$` command gives no response

- Verify `USB_CDC_TxPoll()` is called inside `GCode_Poll()` in `gcode.c`
- Check TX ring is not full (send `?` first to confirm USB is live)
- Confirm `GCode_Send()` calls `USB_CDC_TxWrite()` not `CDC_Transmit_FS` directly

### Position drift over many moves

- Missed steps due to speed too high — reduce `$0=`/`$1=`
- Insufficient motor current — increase driver current limit
- Electrical noise on STEP line — add 100 Ω series resistor on STEP wire
